Key Observations
----------------
1. ADAPTER ACCURACY IS CATASTROPHICALLY LOW (0.14 vs 0.47 for linear).
   Under matched federation_every=2 and n_epochs=5, the feature_adapter
   model fails to learn meaningful representations. This is NOT a
   federation_every confound -- it is a fundamental model training issue.

2. RE-ID AND ENTROPY ARE IDENTICAL between linear and adapter.
   This confirms that Phase A (fingerprinting + concept assignment) is
   model-type-agnostic. The fingerprints are computed from raw data, not
   from model outputs. The concept routing is the same; only the
   downstream model accuracy differs.

3. ADAPTER USES ~4.7x MORE BYTES (5.87M vs 1.24M).
   The feature_adapter model has more parameters (hidden_dim=64,
   adapter_dim=16), so each upload/download costs more. This means the
   adapter is both worse AND more expensive under matched conditions.

4. CFL DOMINATES (0.61 accuracy).
   CFL does not use concept identity tracking (no re-ID or entropy), but
   its clustering-based aggregation achieves the best accuracy by a wide
   margin. This is consistent with prior findings.

5. ORACLE UNDERPERFORMS LINEAR FPT (0.37 vs 0.47).
   Even with perfect concept IDs, the Oracle's per-concept federated
   aggregation hurts on this dataset. This is likely because K=10 clients
   with heterogeneous data cause destructive averaging even within the
   correct concept group -- especially with only 500 samples per step.

Root Cause Analysis
-------------------
The adapter model's failure is not caused by mismatched hyperparameters
(those are now identical). The likely root causes are:

(a) The adapter architecture (64-dim hidden + 16-dim bottleneck) may be
    too expressive for the 64-dim PCA features and 500-sample batches,
    leading to overfitting or unstable gradients.

(b) Even with the zero-init fix, the adapter's additional nonlinearity
    may need different learning rate / epoch tuning than the linear model.
    The shared lr=0.05 and n_epochs=5 were tuned for the linear model.

(c) Federated averaging of adapter parameters may cause more destructive
    interference than averaging linear parameters, because the adapter
    has a nonlinear component where parameter averaging is not equivalent
    to model averaging.

Conclusion
----------
Under matched conditions, the feature_adapter model is strictly worse
than linear on CIFAR-100 recurrence: lower accuracy, higher cost, same
concept tracking. The previous confounded comparison (adapter with
federation_every=5 vs linear with federation_every=2) masked this gap.
Any future adapter experiments should either (a) tune the adapter
hyperparameters independently, or (b) investigate whether the adapter
architecture itself is fundamentally unsuitable for federated averaging.
